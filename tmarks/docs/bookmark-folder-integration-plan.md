## 项目背景

### 当前系统状况

TMarks 目前采用**纯标签系统**进行书签管理，具有以下特点：

**优势：**
- 灵活的多标签关联（一个书签可以有多个标签）
- 强大的标签组合筛选功能
- 智能的相关标签推荐
- 扁平化结构，易于快速访问

**局限性：**
- 缺乏层级结构，难以建立清晰的组织架构
- 书签数量增多后，标签管理变得复杂
- 不符合传统浏览器书签的使用习惯
- 无法直接同步浏览器的文件夹结构

### 用户需求

通过用户反馈和市场调研，我们发现用户有以下核心需求：

1. **结构化组织**：希望像浏览器书签一样，通过文件夹建立清晰的层级结构
2. **习惯延续**：希望保持与浏览器书签管理器相似的使用体验
3. **灵活筛选**：同时需要标签系统提供的灵活筛选能力
4. **无缝同步**：希望能够直接导入浏览器书签，保留原有的文件夹结构
5. **双重分类**：既需要文件夹的"位置"概念，也需要标签的"属性"描述

## 需求分析

### 功能需求

#### 核心需求（P0）

1. **文件夹基础功能**
   - 创建、重命名、删除文件夹
   - 文件夹支持 2-3 层嵌套
   - 书签归属于单一文件夹
   - 文件夹内书签排序

2. **书签管理增强**
   - 书签可以移动到不同文件夹
   - 书签同时保留多标签关联
   - 支持文件夹内搜索和筛选

3. **导航体验**
   - 左侧文件夹树形导航
   - 面包屑显示当前位置
   - 文件夹展开/折叠状态记忆

4. **标签系统保留**
   - 完整保留现有标签功能
   - 标签可以跨文件夹筛选
   - 文件夹 + 标签组合筛选

#### 重要需求（P1）

1. **浏览器书签导入**
   - 支持 Chrome/Edge/Firefox 书签导入
   - 保留原有文件夹层级结构
   - 智能处理标签映射

2. **拖拽操作**
   - 拖拽书签到文件夹
   - 拖拽文件夹调整顺序
   - 拖拽文件夹到其他文件夹（嵌套）

3. **批量操作**
   - 批量移动书签到文件夹
   - 批量添加/移除标签
   - 批量删除

4. **视图切换**
   - 文件夹优先视图
   - 标签优先视图
   - 混合视图（默认）

#### 期望需求（P2）

1. **智能文件夹**
   - 基于规则自动筛选（如：最近添加、未读等）
   - 动态更新内容

2. **文件夹分享**
   - 分享整个文件夹
   - 生成公开链接

3. **文件夹模板**
   - 预设常用文件夹结构
   - 一键创建项目文件夹

### 非功能需求

1. **性能要求**
   - 文件夹树加载时间 < 500ms
   - 书签列表渲染时间 < 1s（1000条）
   - 拖拽操作响应时间 < 100ms

2. **兼容性要求**
   - 向后兼容现有标签系统
   - 现有用户数据平滑迁移
   - 支持桌面端和移动端

3. **可用性要求**
   - 新用户 5 分钟内掌握基本操作
   - 提供清晰的操作引导
   - 支持键盘快捷键

## 竞品分析

### Chrome/Edge 浏览器书签

**设计模式：** 纯文件夹层级

**优点：**
- 简单直观，符合文件系统习惯
- 支持无限层级嵌套
- 拖拽操作流畅
- 与操作系统文件管理器体验一致

**缺点：**
- 没有标签系统，无法横向关联
- 跨文件夹查找困难
- 一个书签只能在一个位置
- 搜索功能较弱

**借鉴点：**
- 文件夹树的交互模式
- 拖拽操作的流畅性
- 右键菜单的功能设计

### Raindrop.io

**设计模式：** Collection（文件夹）+ Tags（标签）双轨制

**优点：**
- Collection 提供结构化组织
- Tags 提供灵活筛选
- 支持 Collection + Tags 组合筛选
- 强大的搜索和过滤功能
- 精美的 UI 设计

**缺点：**
- 学习曲线略高
- 移动端体验一般

**借鉴点：**
- Collection 和 Tags 的协作模式
- 三栏布局设计
- 筛选条件的可视化展示
- 面包屑导航

**核心启示：** 这是最接近我们需求的方案，应作为主要参考对象

### Notion

**设计模式：** 页面层级 + 数据库 + 多维属性

**优点：**
- 极度灵活的组织方式
- 页面可以嵌套
- 数据库支持多种视图
- 强大的筛选和排序

**缺点：**
- 学习曲线陡峭
- 对于简单需求过于复杂
- 性能在大量数据时下降

**借鉴点：**
- 多视图切换的理念
- 面包屑导航设计
- 属性面板的交互

### Evernote

**设计模式：** 笔记本组 > 笔记本 + 全局标签

**优点：**
- 两级文件夹结构清晰
- 全局标签系统强大
- 笔记本 + 标签组合筛选
- 成熟的用户体验

**缺点：**
- 只支持两级嵌套
- 笔记本组功能较弱

**借鉴点：**
- 两级文件夹的限制（避免过深）
- 全局标签的设计
- 侧边栏的折叠展开

### Pinboard

**设计模式：** 扁平化 + 标签 + 特殊标签

**优点：**
- 极简设计
- 标签系统强大
- 支持标签层级（用冒号分隔）

**缺点：**
- 没有真正的文件夹
- 内容多时难以管理

**借鉴点：**
- 标签层级的概念（可选实现）
- 特殊标签的设计思路

### 竞品分析总结

**成功模式的共同点：**
1. 主要组织结构（单一归属）+ 标签系统（多重属性）
2. 文件夹层级控制在 2-4 层
3. 支持文件夹 + 标签组合筛选
4. 提供清晰的当前位置指示
5. 强大的搜索功能

**我们的差异化优势：**
1. 已有成熟的标签系统和用户基础
2. 支持标签页组管理（独特功能）
3. 可以提供更好的浏览器书签同步体验
4. 更现代化的 UI 设计

## 核心设计理念

### 设计原则

#### 1. 单一归属，多重属性

**文件夹 = 主要组织结构（Where）**
- 一个书签只能在一个文件夹中
- 提供清晰的"位置"概念
- 符合传统文件系统习惯

**标签 = 横向关联属性（What）**
- 一个书签可以有多个标签
- 提供灵活的"属性"描述
- 支持跨文件夹的横向关联

**示例：**
```
书签：React 官方文档
位置：📁 工作 > 前端 > React
属性：🏷️ 教程、官方文档、必读
```

#### 2. 渐进式增强

**保持向后兼容：**
- 现有标签系统完全保留
- 用户可以选择不使用文件夹
- 新功能不影响现有工作流

**渐进式采用：**
- 新用户：引导创建文件夹结构
- 老用户：可选择迁移或继续使用标签
- 高级用户：充分利用文件夹 + 标签组合

#### 3. 灵活可配置

**三种视图模式：**
- 混合视图：文件夹 + 标签（默认）
- 文件夹视图：隐藏标签栏，专注文件夹
- 标签视图：隐藏文件夹树，专注标签

**用户可以自由切换：**
- 根据使用场景选择最适合的视图
- 系统记住用户的偏好设置

#### 4. 清晰的信息层级

**筛选优先级：**
```
全局范围（所有书签）
    ↓
文件夹筛选（缩小到特定文件夹及其子文件夹）
    ↓
标签筛选（在文件夹范围内进一步筛选）
    ↓
搜索关键词（在上述范围内精确查找）
```

**可视化展示：**
- 面包屑导航显示当前筛选路径
- 筛选条件可以单独移除
- 一键清空所有筛选

### 核心概念定义

#### 文件夹（Folders）

**特性：**
- 树形结构，支持 2-3 层嵌套
- 每个文件夹有唯一的路径
- 支持颜色和图标自定义
- 可以展开/折叠

**默认文件夹：**
- 全部：虚拟文件夹，显示所有书签
- 未分类：新书签的默认位置
- 回收站：已删除的书签（可恢复）

**限制：**
- 最多 3 层嵌套（避免过深）
- 文件夹名称不能重复（同级）
- 不能删除包含书签的文件夹（需先移动或删除书签）

#### 标签（Tags）

**特性：**
- 扁平结构，全局共享
- 支持颜色自定义
- 显示使用次数
- 支持搜索和排序

**保持不变：**
- 完全保留现有标签功能
- 多标签关联
- 标签组合筛选
- 相关标签推荐

#### 书签（Bookmarks）

**新增属性：**
- folder_id：所属文件夹（单一）
- position：文件夹内的排序位置

**保留属性：**
- tags：关联的标签（多个）
- 其他所有现有属性

## 系统架构设计

### 整体架构

#### 前端架构

**组件层级：**
```
AppShell（应用外壳）
├─ Header（顶部导航栏）
│  ├─ Logo
│  ├─ GlobalSearch（全局搜索）
│  ├─ ViewSwitcher（视图切换）
│  └─ UserMenu（用户菜单）
│
├─ MainContent（主内容区）
│  ├─ FolderSidebar（文件夹侧边栏）- 可折叠
│  │  ├─ FolderTree（文件夹树）
│  │  ├─ FolderActions（文件夹操作）
│  │  └─ QuickAccess（快速访问）
│  │
│  ├─ BookmarkList（书签列表区）
│  │  ├─ Breadcrumb（面包屑导航）
│  │  ├─ FilterBar（筛选条件栏）
│  │  ├─ ActionBar（操作栏）
│  │  ├─ BookmarkGrid/List（书签展示）
│  │  └─ Pagination（分页）
│  │
│  └─ TagSidebar（标签侧边栏）- 可折叠
│     ├─ TagSearch（标签搜索）
│     ├─ TagList（标签列表）
│     └─ SelectedTags（已选标签）
│
└─ MobileBottomNav（移动端底部导航）
```

**状态管理：**
```
全局状态（Zustand）
├─ authStore（认证状态）
├─ themeStore（主题设置）
├─ folderStore（文件夹状态）- 新增
│  ├─ folders（文件夹列表）
│  ├─ selectedFolderId（当前选中）
│  ├─ expandedFolderIds（展开的文件夹）
│  └─ folderViewMode（视图模式）
│
└─ bookmarkStore（书签状态）
   ├─ bookmarks（书签列表）
   ├─ selectedTags（选中的标签）
   ├─ searchKeyword（搜索关键词）
   └─ filterOptions（筛选选项）
```

**数据流：**
```
用户操作
    ↓
组件事件处理
    ↓
Store Action（状态更新）
    ↓
API 调用（后端交互）
    ↓
Store 更新（响应数据）
    ↓
组件重新渲染
```

#### 后端架构

**API 端点设计：**
```
文件夹管理
├─ GET    /api/folders                 获取文件夹列表
├─ POST   /api/folders                 创建文件夹
├─ GET    /api/folders/:id             获取文件夹详情
├─ PUT    /api/folders/:id             更新文件夹
├─ DELETE /api/folders/:id             删除文件夹
├─ POST   /api/folders/:id/move        移动文件夹
└─ GET    /api/folders/:id/bookmarks   获取文件夹内书签

书签管理（扩展）
├─ POST   /api/bookmarks/:id/move      移动书签到文件夹
├─ POST   /api/bookmarks/batch-move    批量移动书签
└─ GET    /api/bookmarks?folder_id=xx  按文件夹筛选

浏览器同步
├─ POST   /api/import/browser-bookmarks 导入浏览器书签
├─ GET    /api/export/browser-format    导出为浏览器格式
└─ POST   /api/sync/browser             同步浏览器书签
```

**数据库设计：**
```
表结构
├─ users（用户表）- 已存在
├─ bookmarks（书签表）- 扩展
│  └─ 新增字段：folder_id, position
├─ tags（标签表）- 保持不变
├─ bookmark_tags（书签标签关联表）- 保持不变
└─ bookmark_folders（文件夹表）- 新增
   ├─ id
   ├─ user_id
   ├─ name
   ├─ parent_id
   ├─ position
   ├─ color
   ├─ icon
   ├─ created_at
   └─ updated_at
```

### 性能优化策略

#### 前端优化

**虚拟滚动：**
- 书签列表使用虚拟滚动（React Virtual）
- 只渲染可见区域的书签
- 支持 10000+ 书签流畅滚动

**懒加载：**
- 文件夹树按需加载子节点
- 书签图片懒加载
- 标签列表分页加载

**缓存策略：**
- React Query 缓存 API 响应
- 文件夹树状态本地缓存
- 用户偏好设置本地存储

**防抖节流：**
- 搜索输入防抖（500ms）
- 拖拽操作节流（100ms）
- 滚动事件节流（200ms）

#### 后端优化

**数据库索引：**
- folder_id 索引（书签表）
- parent_id 索引（文件夹表）
- user_id + folder_id 复合索引

**查询优化：**
- 文件夹树一次性加载（避免 N+1 查询）
- 书签列表分页查询
- 标签统计使用缓存

**缓存策略：**
- 文件夹树缓存（5分钟）
- 标签列表缓存（10分钟）
- 用户偏好缓存（永久，直到更新）

## UI/UX 设计方案

### 桌面端设计

#### 布局方案 A：经典三栏（默认）

**布局比例：** 20% - 55% - 25%

**左侧：文件夹侧边栏（20%）**
- 顶部：新建文件夹按钮
- 中间：文件夹树形列表
  - 支持展开/折叠
  - 显示书签数量
  - 拖拽排序
  - 右键菜单
- 底部：回收站入口

**中间：书签列表区（55%）**
- 顶部：面包屑导航 + 筛选条件
- 操作栏：搜索、排序、视图切换、批量操作、新增
- 书签展示：列表/卡片/极简/标题视图
- 底部：分页控制

**右侧：标签侧边栏（25%）**
- 顶部：标签搜索 + 排序 + 布局切换
- 中间：标签列表
  - 已选标签高亮
  - 相关标签提示
  - 显示使用次数
- 底部：清空选择按钮

**特点：**
- 信息密度高，适合专业用户
- 文件夹和标签同时可见
- 支持快速组合筛选

#### 布局方案 B：文件夹优先（可选）

**布局比例：** 25% - 75%

**左侧：文件夹侧边栏（25%）**
- 同方案 A

**右侧：书签列表区（75%）**
- 标签侧边栏隐藏
- 书签卡片更大，显示更多信息
- 书签上显示关联的标签
- 顶部有"显示标签栏"按钮

**特点：**
- 专注于文件夹导航
- 适合文件夹结构清晰的用户
- 更大的书签展示空间

#### 布局方案 C：标签优先（可选）

**布局比例：** 70% - 30%

**左侧：书签列表区（70%）**
- 文件夹树隐藏
- 顶部有"显示文件夹"按钮（点击弹出抽屉）
- 书签上显示所属文件夹路径

**右侧：标签侧边栏（30%）**
- 同方案 A

**特点：**
- 专注于标签筛选
- 适合习惯标签系统的老用户
- 保持原有使用习惯

### 移动端设计

#### 主界面

**顶部栏：**
- 左侧：菜单按钮（打开文件夹抽屉）

- 📚 书签（当前页）
- 📁 文件夹（打开文件夹抽屉）

### 交互设计

#### 文件夹操作

**创建文件夹：**
1. 点击"新建文件夹"按钮
2. 弹出对话框，输入名称
3. 可选：选择父文件夹、颜色、图标

## 浏览器书签同步方案

### 需求分析

#### 用户场景

**场景 1：首次导入**

- 避免重复

**场景 3：双向同步（未来）**
- TMarks 与浏览器实时同步
- 通过浏览器扩展实现
- 自动检测变化并同步

#### 技术挑战

**文件格式差异：**
- Chrome/Edge：HTML 格式（Netscape Bookmark File Format）
- Firefox：JSON 格式
- Safari：plist 格式

**数据映射：**
- 浏览器文件夹 → TMarks 文件夹
- 浏览器书签 → TMarks 书签
- 如何处理标签？（浏览器没有标签概念）

**冲突处理：**
- 同名文件夹如何处理？
- 重复书签如何识别？
- 修改冲突如何解决？

### 导入方案设计

#### 支持的格式

**Chrome/Edge HTML 格式：**
```html
<!DOCTYPE NETSCAPE-Bookmark-file-1>
<META HTTP-EQUIV="Content-Type" CONTENT="text/html; charset=UTF-8">
<TITLE>Bookmarks</TITLE>
<H1>Bookmarks</H1>
<DL><p>
    <DT><H3>工作</H3>
    <DL><p>
        <DT><H3>前端</H3>
        <DL><p>
            <DT><A HREF="https://react.dev" ADD_DATE="1234567890">React</A>
        </DL><p>
    </DL><p>
</DL><p>
```

**Firefox JSON 格式：**
```json
{
  "title": "Bookmarks",
  "children": [
    {
      "title": "工作",
      "type": "folder",
      "children": [
        {
          "title": "React",
          "type": "bookmark",
          "url": "https://react.dev"
        }
      ]
    }
  ]
}
```

#### 解析策略

**HTML 解析：**
1. 使用 DOMParser 解析 HTML
2. 递归遍历 DL/DT 标签
3. H3 标签 = 文件夹
4. A 标签 = 书签
5. 保留层级关系

**JSON 解析：**
1. 直接解析 JSON 对象
2. 递归遍历 children 数组
3. type: "folder" = 文件夹
4. type: "bookmark" = 书签
5. 保留层级关系

#### 导入流程

**步骤 1：文件上传**
- 用户选择书签文件
- 前端验证文件格式
- 显示文件信息（书签数量、文件夹数量）

**步骤 2：预览和配置**
- 显示文件夹树预览
- 用户选择导入选项：
  - 导入到根目录 / 导入到指定文件夹
  - 保留文件夹结构 / 扁平化导入
  - 自动生成标签（基于文件夹名称）
  - 处理重复书签：跳过 / 覆盖 / 保留两者

**步骤 3：执行导入**
- 后端解析文件
- 创建文件夹结构
- 批量创建书签
- 建立关联关系
- 显示导入进度

**步骤 4：导入结果**
- 显示导入统计：
  - 成功导入：X 个文件夹，Y 个书签
  - 跳过重复：Z 个
  - 失败：W 个（显示原因）
- 提供"查看导入的书签"链接

#### 重复检测策略

**书签去重规则：**
1. **精确匹配**：URL 完全相同
2. **规范化匹配**：URL 规范化后相同
   - 移除 trailing slash
   - 统一协议（http/https）
   - 移除 URL 参数（可选）
3. **标题 + URL 匹配**：标题和 URL 都相同

**处理策略：**
- **跳过**：保留现有书签，不导入重复项
- **覆盖**：用导入的书签覆盖现有书签
- **保留两者**：都保留，添加后缀区分
- **合并**：合并标签和属性

#### 文件夹映射策略

**场景 1：首次导入（无现有文件夹）**
- 直接创建浏览器的文件夹结构
- 保持原有层级关系

**场景 2：已有文件夹结构**
- **同名文件夹**：
  - 选项 A：合并到现有文件夹
  - 选项 B：创建新文件夹（添加后缀）
  - 选项 C：用户手动选择映射关系

**场景 3：特殊文件夹处理**
- 浏览器"书签栏" → TMarks "收藏夹"（特殊文件夹）
- 浏览器"其他书签" → TMarks "未分类"
- 浏览器"移动设备书签" → TMarks "移动书签"（新建）

#### 标签生成策略

**自动生成标签（可选功能）：**

**策略 1：基于文件夹路径**
- 文件夹路径：工作 > 前端 > React
- 生成标签：工作、前端、React
- 优点：保留层级信息
- 缺点：标签可能过多

**策略 2：基于顶级文件夹**
- 只使用顶级文件夹名称作为标签
- 文件夹路径：工作 > 前端 > React
- 生成标签：工作
- 优点：标签数量可控
- 缺点：丢失细节信息

**策略 3：智能提取**
- 分析书签标题和 URL
- 提取关键词作为标签
- 例如：URL 包含 "github.com" → 标签 "GitHub"
- 优点：标签更有意义
- 缺点：需要复杂的算法

**用户选择：**
- 导入时提供选项
- 可以预览生成的标签
- 可以手动调整

### 数据迁移脚本

**步骤 1：创建文件夹表**
```sql
CREATE TABLE bookmark_folders (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  name TEXT NOT NULL,
  parent_id TEXT,

  icon TEXT,

  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (parent_id) REFERENCES bookmark_folders(id) ON DELETE CASCADE

CREATE INDEX idx_folders_user ON bookmark_folders(user_id);

```

**步骤 2：扩展书签表**
```sql
ALTER TABLE bookmarks ADD COLUMN folder_id TEXT;
ALTER TABLE bookmarks ADD COLUMN position INTEGER DEFAULT 0;

CREATE INDEX idx_bookmarks_folder ON bookmarks(folder_id);
CREATE INDEX idx_bookmarks_user_folder ON bookmarks(user_id, folder_id);
```

**步骤 3：为每个用户创建系统文件夹**
```sql

INSERT INTO bookmark_folders (id, user_id, name, parent_id, position, icon, is_system, created_at, updated_at)

  '未分类',
  NULL,
  0,

  1,
  datetime('now'),
  datetime('now')

-- 为每个现有用户创建"回收站"文件夹

  NULL,
  999,
  '🗑️',
  1,
  datetime('now'),
  datetime('now')
```

**步骤 4：将现有书签关联到"未分类"文件夹**
```sql
UPDATE bookmarks
SET folder_id = 'system-unsorted-' || user_id
WHERE folder_id IS NULL;
```

**步骤 5：更新书签位置**
```sql
-- 为每个文件夹内的书签分配位置
WITH ranked_bookmarks AS (
  SELECT 
    id,
    ROW_NUMBER() OVER (PARTITION BY folder_id ORDER BY created_at) - 1 AS new_position
  FROM bookmarks
)
UPDATE bookmarks
SET position = (SELECT new_position FROM ranked_bookmarks WHERE ranked_bookmarks.id = bookmarks.id);
```

## 用户迁移策略

### 迁移原则

**核心理念：** 让用户自主选择和重构

**设计思路：**
- 不进行自动迁移
- 现有书签保持原状（无文件夹归属）
- 用户可以自由选择使用方式
- 提供便捷的整理工具

### 数据库迁移

**数据库层面的变更：**

1. **创建文件夹表**
   - 新增 bookmark_folders 表
   - 不预先创建任何文件夹（包括系统文件夹）

2. **扩展书签表**
   - 添加 folder_id 字段（允许 NULL）
   - 添加 position 字段（默认 0）
   - 现有书签的 folder_id 保持为 NULL

3. **保持标签系统不变**
   - 标签表不做任何修改
   - 书签-标签关联表不做任何修改
   - 所有标签关系完整保留

**结果：**
- 现有用户的书签完全不受影响
- folder_id 为 NULL 表示书签未归入任何文件夹
- 用户可以继续使用纯标签模式

## 风险评估与应对

### 技术风险

#### 风险 1：性能问题

**风险描述：**
- 文件夹树查询可能产生 N+1 问题
- 大量书签时，递归查询子文件夹性能下降
- 拖拽操作频繁时，数据库写入压力大

**应对措施：**
- 使用 CTE（公共表表达式）优化递归查询
- 添加适当的数据库索引
- 实现查询结果缓存
- 前端使用虚拟滚动
- 拖拽操作使用防抖和节流

**监控指标：**
- 文件夹树加载时间 < 500ms
- 书签列表渲染时间 < 1s
- 拖拽操作响应时间 < 100ms

#### 风险 2：数据一致性

**风险描述：**
- 删除文件夹时，书签的 folder_id 可能变成孤儿引用
- 并发操作可能导致数据不一致
- 文件夹嵌套可能形成循环引用

**应对措施：**
- 使用外键约束（ON DELETE SET NULL）
- 删除文件夹前检查是否包含书签
- 使用数据库事务保证原子性
- 前端验证文件夹嵌套深度
- 后端检测循环引用

**数据保护：**
- 定期数据库备份
- 提供数据导出功能
- 实现软删除（回收站）

#### 风险 3：浏览器兼容性

**风险描述：**
- 不同浏览器的书签格式可能有差异
- 特殊字符处理不当可能导致解析失败
- 大文件上传可能超时

**应对措施：**
- 充分测试各种浏览器导出的书签文件
- 实现健壮的错误处理
- 支持分批导入大文件
- 提供详细的错误提示

### 用户体验风险

#### 风险 1：学习曲线

**风险描述：**
- 用户可能不理解文件夹和标签的区别
- 混合模式可能让用户困惑
- 界面复杂度增加

**应对措施：**
- 提供清晰的功能引导
- 提供视频教程和文档
- 设计直观的 UI
- 提供多种视图模式供选择
- 收集用户反馈，持续优化

#### 风险 2：功能冗余

**风险描述：**
- 文件夹和标签功能重叠
- 用户可能不知道该用哪个
- 可能导致混乱的组织结构

**应对措施：**
- 明确文件夹和标签的定位
- 提供最佳实践建议
- 提供整理建议功能
- 允许用户选择单一模式

#### 风险 3：迁移阻力

**风险描述：**
- 现有用户可能不愿意改变习惯
- 整理大量书签需要时间
- 可能导致用户流失

**应对措施：**
